# [01] Первая программа: syscall

> **Трек:** Assembly · **Уровень:** 01 · **Сложность:** ★☆☆☆☆

## 1. Какую проблему решаем?

Мы хотим, чтобы компьютер **что-то сделал**: завершился с кодом 0, напечатал строчку, прочитал файл.

Но процессор не умеет «завершиться» или «напечатать». Он умеет только выполнять инструкции: сложить два числа, записать в память, перейти на другой адрес.

Любое взаимодействие с внешним миром (вывод на экран, чтение файла, завершение программы) делает **ядро операционной системы** (Linux). Программа может попросить ядро о чём-то через **системный вызов** — `syscall`.

---

## 2. Что такое файл? Что такое .asm?

Файл — это просто именованная последовательность байтов на диске. Текстовый файл — те же байты, но каждый байт — это код символа.

Файл с расширением `.asm` — это текстовый файл, содержащий инструкции на языке ассемблера. Ты можешь создать его любым текстовым редактором (nano, vim, vs code).

```
exit.asm           ← имя файла
│
расширение .asm    ← говорит нам и системе: «это код на ассемблере»
```

## 3. Структура программы на ассемблере

Любая программа на ассемблере (в нашем курсе) состоит из:

```assembly
section .text       ; ← секция кода — сюда пишем инструкции
    global _start   ; ← говорим линкеру: «вот точка входа»

_start:             ; ← метка (label) — имя для адреса
    ...инструкции... ; ← то, что будет выполнять процессор
```

**section .text** — объявляет секцию кода. Всё, что идёт после неё до следующей секции, будет машинным кодом.

**global _start** — делает метку `_start` видимой для линкера. Без этого ld не найдёт, с чего начинать программу.

**_start:** — метка. Метка — это имя для адреса в памяти. Где будет находиться эта инструкция в памяти — туда и указывает `_start`. Линкер знает: программа начинается с `_start`.

## 4. Первая программа: exit(0)

Напишем программу, которая просто завершается с кодом 0. Это минимальная программа, которая вообще что-то делает.

```assembly
; exit.asm — завершиться с кодом 0
section .text
    global _start

_start:
    mov rax, 60     ; rax = 60 (номер syscall exit)
    xor rdi, rdi    ; rdi = 0 (код возврата, 0 = успех)
    syscall         ; передать управление ядру
```

### Разбор каждой строчки

**`mov rax, 60`**
- `mov` — инструкция «move» (положить значение)
- `rax` — куда кладём (регистр RAX)
- `60` — что кладём (число 60)
- Результат: в регистре RAX теперь число 60

**`xor rdi, rdi`**
- `xor` — инструкция «исключающее ИЛИ»
- `rdi, rdi` — rdi xor rdi. Любое число xor само с собой = 0
- Это короткий способ записать «положить 0 в rdi»
- Можно было написать `mov rdi, 0`, но `xor rdi, rdi` короче и быстрее
- Результат: в регистре RDI теперь 0

**`syscall`**
- Одна инструкция без операндов
- Передаёт управление ядру Linux
- Ядро смотрит на RAX (60) и понимает: «нужно выполнить exit»
- Берёт аргументы из регистров: RDI = 0 (код возврата)
- Завершает программу

### Почему rax=60 и rdi=0?

В Linux каждый системный вызов имеет номер:
- `exit` — номер 60
- `write` — номер 1
- `read` — номер 0
- `open` — номер 2
- и т.д.

Аргументы syscall передаются в регистрах в строгом порядке:
```
rax  → номер syscall
rdi  → первый аргумент
rsi  → второй аргумент
rdx  → третий аргумент
r10  → четвёртый аргумент
r8   → пятый аргумент
r9   → шестой аргумент
```

Для `exit(код_возврата)`:
- Номер в rax: 60
- Первый аргумент (код возврата) в rdi: 0

## 5. Как собрать и запустить

Создай файл `exit.asm` с текстом выше. Потом:

```bash
# Шаг 1: Ассемблирование
# -f elf64 — формат объектного файла (64-битный Linux)
# exit.asm — исходный файл
# -o exit.o — выходной файл
nasm -f elf64 exit.asm -o exit.o

# Шаг 2: Линковка
# exit.o — объектный файл
# -o exit — имя выходного исполняемого файла
ld exit.o -o exit

# Шаг 3: Запуск
./exit

# Шаг 4: Проверить код возврата
echo $?    → выведет 0
```

Если всё правильно, `echo $?` покажет 0. Если нет — будет ошибка.

## 6. Вторая программа: hello world

Теперь программа, которая выводит текст. Для этого используем syscall `write` (номер 1).

```assembly
; hello.asm — вывести "Hello!" и завершиться
section .data           ; ← секция данных
    msg db "Hello!", 10 ; ← msg = адрес строки "Hello!\n"
                        ;    db = define byte (выделить байты)
                        ;    "Hello!" = 6 байт: 72, 101, 108, 108, 111, 33
                        ;    10 = перевод строки ('\n')
    len equ $ - msg     ; ← len = длина строки (текущий адрес - адрес msg)

section .text
    global _start

_start:
    ; write(1, msg, len)
    mov rax, 1          ; rax = 1  (номер syscall write)
    mov rdi, 1          ; rdi = 1  (первый аргумент: stdout)
    mov rsi, msg        ; rsi = адрес строки msg (второй аргумент)
    mov rdx, len        ; rdx = длина строки len (третий аргумент)
    syscall             ; вызвать ядро

    ; exit(0)
    mov rax, 60         ; rax = 60 (номер syscall exit)
    xor rdi, rdi        ; rdi = 0  (код возврата)
    syscall             ; вызвать ядро
```

### Разбор секции .data

**section .data** — объявляет секцию данных. Сюда мы кладём переменные, строки, массивы — всё, что должно существовать в памяти во время работы программы.

**`msg db "Hello!", 10`**

Разберём по кусочкам:

| Часть | Что значит |
|-------|-----------|
| `msg` | **Метка** (label). Имя для адреса в памяти. Когда мы пишем `msg`, ассемблер подставляет адрес первого байта этой строки. Можно называть как угодно: `msg`, `hello`, `my_string`, `x` — главное, чтобы имя было понятным |
| `db` | **Define Byte**. Директива ассемблера. Говорит: «выдели байты и заполни их вот этими значениями». `db` = 1 байт на элемент. Есть ещё `dw` (2 байта), `dd` (4 байта), `dq` (8 байт) |
| `"Hello!"` | **Строковый литерал**. 6 символов: H(72) e(101) l(108) l(108) o(111) !(33). Ассемблер превращает каждый символ в его ASCII-код и выделяет по одному байту |
| `, 10` | **Ещё один байт**. Запятая разделяет элементы. 10 — это код перевода строки (`\n`). Можно было написать `"Hello!", 10` или `"Hello!\n"` — эффект тот же |

Итого: `msg db "Hello!", 10` выделяет 7 байт в памяти и заполняет их:
```
Адрес:  msg → [72] [101] [108] [108] [111] [33] [10]
Символ:     H     e     l     l     o     !    \n
```

**`len equ $ - msg`**

| Часть | Что значит |
|-------|-----------|
| `len` | Имя константы |
| `equ` | **Equate**. Приравнять. Не выделяет память, просто вычисляет значение. Где бы мы ни написали `len`, ассемблер подставит число |
| `$` | **Текущий адрес**. В данном случае — адрес сразу после последнего байта строки |
| `msg` | Адрес первого байта строки |
| `$ - msg` | Разница адресов = длина строки в байтах |

Вычисление: если `msg` находится по адресу 0x100, а строка занимает 7 байт, то `$` = 0x107, и `len` = 0x107 - 0x100 = 7.

Почему это «самовычисляющаяся длина»? Потому что если мы изменим строку, `len` пересчитается автоматически. Не нужно вручную считать байты.

### Разбор секции .text

```assembly
mov rax, 1     ; rax = 1. Номер syscall write
mov rdi, 1     ; rdi = 1. Первый аргумент write: файловый дескриптор
mov rsi, msg   ; rsi = адрес строки. Второй аргумент: буфер
mov rdx, len   ; rdx = длина. Третий аргумент: количество байт
syscall        ; выполнить write(1, msg, len)
```

**Файловый дескриптор 1 (stdout)**
- 0 = stdin (ввод с клавиатуры)
- 1 = stdout (вывод на экран)
- 2 = stderr (вывод ошибок)

Когда Linux запускает программу, он открывает три потока: stdin, stdout, stderr. `write(1, ...)` пишет в stdout — то есть на экран.

### Что видит ядро, когда мы вызываем syscall

```
Регистры перед syscall:
  rax = 1    → «хочу write»
  rdi = 1    → «в stdout»
  rsi = msg  → «вот адрес данных»
  rdx = 7    → «вот сколько байт»

Ядро (упрощённо):
  1. Смотрит rax: 1 → это write
  2. Берёт rdi: 1 → файловый дескриптор
  3. Берёт rsi: адрес → откуда читать
  4. Берёт rdx: 7 → сколько байт
  5. Копирует 7 байт из памяти по адресу rsi в stdout
  6. Возвращает в rax количество записанных байт (7)
  7. Передаёт управление обратно в программу (на следующую инструкцию)
```

После `syscall` программа продолжает выполнение со следующей инструкции:

```assembly
mov rax, 60    ; готовим exit
xor rdi, rdi   ; код возврата 0
syscall        ; завершаемся
```

## 8. Частые ошибки

### Ошибка 1: "No such file or directory" при запуске nasm
```
bash: nasm: command not found
```
**Решение:** Установи nasm: `sudo apt install nasm`

### Ошибка 2: "undefined reference to _start"
```
ld: warning: cannot find entry symbol _start; not setting start address
```
**Причина:** Забыли `global _start` или метка называется иначе.
**Решение:** Проверь, что в коде есть `global _start` и `_start:`.

### Ошибка 3: "segmentation fault" при запуске
```
./hello
Segmentation fault
```
**Причина:** Программа не завершилась через exit. Процессор дошёл до конца кода и продолжил выполнять мусор.

**Решение:** Убедись, что в конце есть `mov rax, 60; xor rdi, rdi; syscall`.

### Ошибка 4: ничего не выводится
```
./hello
```
(пусто)

**Причина:** Неверная длина строки. Write записал 0 байт или не те байты.
**Решение:** Проверь `len equ $ - msg`. Если строка после db — убедись, что len идёт сразу после неё.

## 9. Упражнения

1. **Код возврата.** Измени `exit(0)` на `exit(42)`. Собери, запусти, проверь `echo $?`. Должно быть 42.

2. **Другая строка.** Измени "Hello!" на "Xodex OS". Не забудь, что длина изменится (но `len equ $ - msg` пересчитает автоматически). Проверь, что выводится правильно.

3. **Два вывода.** Напиши программу, которая выводит две строки: "Hello" и "World" на отдельных строках. Подсказка: понадобится дважды вызвать write.

4. **Без .data.** Напиши программу, которая выводит строку, не используя секцию `.data`. Строку можно разместить прямо в `.text`, рядом с инструкциями. (Это работает, потому что код тоже находится в памяти.)

   ```assembly
   section .text
       global _start
   _start:
       ; напиши hello прямо здесь
       ; подсказка: нужна метка на строку
   ```
